# Техническое задание: realtime-проверка сигналов и минимизация latency в `arbitration-trader`

## 1. Цель

Нужно доработать `arbitration-trader` так, чтобы проверка условий на открытие и закрытие сделок выполнялась в realtime при поступлении новых данных по сокету, а задержка между обнаружением сигнала и фактической отправкой ордеров была минимальной.

Ключевой приоритет:

- минимальная задержка между новым состоянием orderbook и решением на `open` / `close`;
- отсутствие polling-подхода для entry/profit/liquidation сигналов;
- сохранение защитной логики against unmanaged exposure, partial fills, close sync failures и reconciliation mismatch.

## 2. Что считается нормальным и не требует изменения

Следующее поведение допустимо и не является проблемой в рамках данного ТЗ:

- timeout-close может и дальше проверяться интервалом;
- watchdog на timeout раз в `10s` остается отдельным risk-control механизмом;
- background retry-механики для `pendingCloseSync`, cleanup и reconciliation могут оставаться интервальными, если они не отвечают за realtime entry/exit сигнал.

Это ТЗ относится именно к:

- открытию сделок по spread signal;
- закрытию сделок по profit signal;
- закрытию сделок по liquidation/drawdown trigger.

## 3. Проблема

Система должна реагировать на рыночный сигнал по факту обновления данных с биржевых сокетов, а не по периодическому циклу или отложенному обслуживанию.

Требуется добиться следующего поведения:

1. При каждом релевантном обновлении orderbook с любой из двух бирж должна инициироваться актуальная проверка состояния symbol.
2. При выполнении условий входа решение на открытие должно приниматься немедленно, без искусственных задержек.
3. При выполнении условий profit-close или liquidation-close решение на закрытие должно приниматься немедленно, без ожидания interval tick.
4. Все дополнительные операции в критическом пути должны быть либо:
   - строго необходимыми для безопасности;
   - либо вынесенными из latency-critical path.

## 4. Целевое поведение

### 4.1. Realtime event-driven модель

Для каждого symbol проверка должна запускаться реактивно по socket updates:

1. обновилась книга на primary exchange;
2. обновилась книга на secondary exchange;
3. система получает согласованное актуальное состояние книги для symbol;
4. immediately запускается проверка:
   - entry, если позиции нет;
   - exit, если позиция есть.

Проверка не должна зависеть от периодического таймера для entry/profit/liquidation сценариев.

### 4.2. Минимизация latency

Критический путь от рыночного сигнала до отправки ордера должен быть максимально коротким.

Требование:

- не добавлять искусственные sleep/debounce/throttle перед `open` и `close`, кроме случаев, где это нужно для exchange consistency и явно обосновано;
- минимизировать число blocking-операций перед вызовом `createMarketOrder`;
- убрать из критического пути все вычисления и проверки, которые можно безопасно подготовить заранее или поддерживать в hot cache.

### 4.3. Границы допустимых компромиссов

Ускорение не должно ломать:

- проверку реального состояния позиций;
- защиту от повторного входа;
- reduce-only semantics на close / rollback;
- risk lock при cleanup/reconciliation проблемах;
- корректность `pendingCloseSync`;
- корректность execution journal.

Если какой-либо safety-check мешает latency, его нельзя просто удалить. Нужна либо:

- эквивалентная по надежности замена;
- либо перенос проверки из критического пути в более ранний этап.

## 5. Объем работ

## 5.1. Что входит

1. Анализ полного пути:
   - socket update -> signal evaluation -> `executeOpen`;
   - socket update -> exit evaluation -> `executeClose`.
2. Выявление всех blocking steps в критическом пути.
3. Проектирование event-driven схемы, где entry/profit/liquidation принимаются по realtime updates.
4. Минимизация latency между:
   - обнаружением сигнала и стартом отправки ордеров;
   - обнаружением close signal и стартом отправки close orders.
5. Добавление измерений latency и технических метрик.
6. Подготовка безопасной стратегии миграции без деградации risk-controls.

## 5.2. Что не входит

1. Изменение timeout-watchdog semantics.
2. Ослабление safety-checks без компенсационного механизма.
3. Переписывание всей архитектуры orderbook providers, если цель можно достигнуть локальными изменениями в runtime path.
4. Изменение торговой формулы spread/PnL, если это не требуется именно для latency.

## 6. Функциональные требования

### 6.1. Entry

1. Entry должен проверяться по каждому релевантному socket update.
2. Решение на entry должно использовать самые свежие доступные данные обеих книг.
3. После обнаружения entry signal система должна начинать путь к отправке ордеров немедленно.
4. Если для symbol уже идет обработка, дублирующий запуск не должен создавать вторую конкурентную попытку открытия.

### 6.2. Exit

1. Profit-close должен проверяться по каждому релевантному socket update.
2. Liquidation/drawdown protection должна проверяться по каждому релевантному socket update.
3. Если условие close выполнено, закрытие должно стартовать без ожидания interval tick.
4. Если часть close уже выполнена, система должна продолжать корректный partial-close flow без повторного открытия позиции.

### 6.3. Concurrency и ordering

1. Для одного symbol одновременно может идти только один active decision path.
2. Новые book updates не должны запускать параллельные `open`/`close` операции по одному symbol.
3. При высокой частоте обновлений система не должна бесконтрольно накапливать backlog задач.
4. Если во время обработки пришли новые данные, после завершения текущей обработки symbol должен быть переоценен на самом свежем состоянии.

### 6.4. Freshness данных

1. Решение не должно приниматься по заведомо устаревшей комбинации двух книг.
2. Нужен явный критерий актуальности snapshot pair:
   - допустимый age книги;
   - допустимый skew между primary и secondary snapshot timestamps.
3. При нарушении freshness сигнал не исполняется.

## 7. Требования к latency

Нужно ввести измеряемые KPI на прод-пути.

Минимальный набор метрик:

1. `socket_update_to_check_start_ms`
2. `check_start_to_signal_detected_ms`
3. `signal_detected_to_order_submit_start_ms`
4. `close_signal_detected_to_close_submit_start_ms`
5. `order_submit_start_to_exchange_ack_ms`
6. `full_close_sync_duration_ms`

Целевой принцип:

- критический путь должен измеряться end-to-end;
- любые всплески latency должны быть видны в логах/метриках;
- P95/P99 latency по сигналам должна быть наблюдаемой.

Числовые SLO нужно зафиксировать после базового профилирования текущей реализации, но архитектура должна проектироваться под минимально возможный latency, а не под best-effort polling.

## 8. Технические требования к реализации

### 8.1. Критический путь

Нужно разделить операции на две группы:

1. latency-critical:
   - получение актуального состояния книги;
   - расчет spread / VWAP / PnL;
   - проверка порогов;
   - запуск отправки ордеров.
2. non-critical:
   - тяжелые диагностические логи;
   - вторичные вычисления;
   - операции, которые можно делать заранее;
   - операции, которые можно асинхронно записать без блокировки order submission.

### 8.2. Предварительная подготовка

Нужно проанализировать, какие данные можно держать в горячем состоянии заранее:

- нормализованные market constraints;
- precomputed trade amount / rounded size per symbol, если это допустимо;
- свежий локальный state позиций;
- readiness-флаги по symbol;
- freshness timestamps для обеих книг.

### 8.3. Проверки позиций

Проверки, связанные с фактическими позициями на бирже, сохраняются как safety requirement.

Требование:

- проанализировать, какие из них обязаны оставаться синхронными прямо перед `open`/`close`;
- какие можно перевести в непрерывно поддерживаемое локальное состояние;
- какие можно подтвердить заранее;
- какие можно выполнять только при подозрении на mismatch.

Недопустимо просто убрать проверку существующих позиций перед входом без безопасной замены.

### 8.4. Execution journal

Execution journal должен сохранять защитную роль, но не должен неоправданно раздувать latency критического пути.

Нужно определить:

- какие записи обязательны до отправки ордеров;
- какие можно писать после старта ордеров;
- какие можно писать асинхронно с гарантией целостности.

### 8.5. Логирование и наблюдаемость

Нужно добавить структурированное измерение этапов open/close pipeline:

- timestamp получения market event;
- timestamp начала проверки;
- timestamp принятия решения;
- timestamp начала отправки primary/secondary order;
- timestamp получения exchange result;
- timestamp завершения Django sync.

## 9. Нефункциональные требования

1. Изменения не должны ухудшить корректность recovery после рестарта.
2. Изменения не должны увеличивать вероятность duplicated execution.
3. Изменения не должны создавать unbounded queue growth при высокой частоте сокетов.
4. Изменения не должны ухудшить поведение при temporary socket desync.
5. Изменения должны быть совместимы с текущими risk-lock, cleanup и close-sync механизмами.

## 10. Ограничения и риски

### 10.1. Основные риски

1. Ускорение critical path может привести к пропуску safety checks.
2. Сильное упрощение pre-open verification может привести к входу поверх уже существующей позиции.
3. Агрессивная реакция на каждый tick может увеличить нагрузку на CPU и REST API.
4. Несогласованные timestamp двух бирж могут давать ложные сигналы.
5. Слишком ранний close без подтверждения актуального состояния может ломать idempotency partial-close flow.

### 10.2. Обязательные меры защиты

1. Сохранить per-symbol mutex / serialization.
2. Сохранить protection от duplicate open/close.
3. Сохранить risk lock при reconciliation mismatch.
4. Сохранить retry-логику при pending close sync.
5. Явно валидировать свежесть orderbook snapshots.

## 11. Критерии приемки

Задача считается выполненной, если:

1. Entry signal проверяется event-driven по socket updates, а не по периодическому циклу.
2. Profit-close и liquidation-close проверяются event-driven по socket updates, а не по периодическому циклу.
3. Timeout-close может оставаться interval-based.
4. Для каждого symbol исключены конкурентные дублирующие execution path.
5. Добавлены измеримые latency-метрики по пути signal -> submit.
6. Под высокой частотой сокетов нет бесконтрольного накопления backlog по symbol.
7. Risk-controls, cleanup, recovery и close sync продолжают работать корректно.
8. Появляется документированное сравнение:
   - текущий latency path;
   - новый latency path;
   - какие этапы были убраны, сокращены или вынесены из critical path.

## 12. План проектирования

Реализация должна начинаться не с кода, а с технического разбора текущего пути.

Нужно подготовить:

1. карту текущего critical path для `open`;
2. карту текущего critical path для `close`;
3. список всех синхронных blocking operations;
4. предложение по новой event-driven схеме обработки symbol;
5. план миграции без потери safety guarantees;
6. список метрик и логов для профилирования до и после изменений.

## 13. Ожидаемый результат

После реализации система должна вести себя так:

1. При новом socket update по symbol runtime немедленно переоценивает вход/выход.
2. При наличии valid signal решение на сделку не ждет interval tick.
3. Между обнаружением сигнала и стартом отправки ордеров остается только минимально необходимый набор операций.
4. Safety-поведение сохраняется: runtime не торгует в неясном состоянии и блокирует новые входы при risk/incidents.
