# Правила разработки

## Качество кода

- Пиши код как **сеньор-разработчик** — чистый, читаемый, поддерживаемый.
- Следуй принципам **SOLID**, **DRY**, **KISS**, **YAGNI**.
- Применяй лучшие практики архитектуры: слоистая архитектура, разделение ответственности, dependency injection.
- Не допускай "quick & dirty" решений. Каждое решение должно быть **production-ready**.
- Покрывай edge-cases, обрабатывай ошибки корректно.
- Используй типизацию везде, где это возможно (TypeScript strict mode, Python type hints).

## Angular

- **НИКОГДА** не пиши HTML и SCSS инлайном в `.ts` файлах компонентов.
- Всегда используй отдельные файлы: `component.html` и `component.scss`.
- Используй `templateUrl` и `styleUrl` вместо `template` и `styles`.

## Язык общения

- Планирование, обсуждение архитектуры и ответы — **на русском языке**.
- Комментарии в коде и документация — на английском.

## Документация

- **Обязательно** веди документацию проекта и поддерживай её в актуальном состоянии.
- При любых изменениях в архитектуре, API, моделях, конфигурации — **немедленно обновляй** соответствующую документацию.
- Документация должна включать:
  - **README.md** — общее описание проекта, инструкции по запуску, стек технологий.
  - **Архитектурные решения** — описание структуры, слоёв, взаимодействия компонентов.
  - **API-документация** — эндпоинты, параметры, примеры запросов/ответов.
  - **Changelog** — журнал значимых изменений.
- Не откладывай документирование на потом — оно идёт **параллельно с разработкой**.

## Выполнение команд

- **Не выполняй команды в терминале самостоятельно.**
- Вместо этого — выписывай команды, которые нужно выполнить, и пользователь выполнит их сам.
- Формат: блок кода с командой и краткое пояснение, что она делает.
